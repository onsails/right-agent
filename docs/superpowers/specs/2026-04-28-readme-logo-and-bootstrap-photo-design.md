# README logo + bootstrap welcome photo

**Date:** 2026-04-28
**Status:** Design — pending implementation plan

## Context

`docs/brand-guidelines.html` defines a full logo system (primary mark, character C1 with eyes/smile, six-variant wordmark family, lockups, social presets). On disk, none of those assets exist yet — only `images/screenshot.png`. The README currently opens with `# Right Agent`, badges, tagline, and the screenshot; there is no logo.

The Telegram-side bot bootstrap flow (`BOOTSTRAP.md` present in the agent dir → BOOTSTRAP_INSTRUCTIONS injected into the system prompt → CC asks the user for name/vibe/etc. → writes IDENTITY.md / SOUL.md / USER.md → BOOTSTRAP.md is removed) has no visual signature. The agent's first reply is plain text.

## Goals

1. Show the canonical brand-guide README header at the top of `README.md` instead of a plain H1.
2. On the bot's very first reply during bootstrap, attach a "lively" character-mark photo so the first agent appearance has a visual identity.

## Non-goals

- Building out the full `assets/` manifest from brand guide section 11 (favicon, og-image, lockup-stacked, every wordmark variant, etc.). Only the two assets that are needed by this change ship.
- Animated stickers, GIFs, Lottie — the welcome photo is a static PNG.
- Build-time SVG→PNG rendering (no `resvg` / `rsvg-convert` dependency in the build graph). PNGs are committed to the repo, regenerated by hand when the source SVG changes.
- Agent prompt changes. BOOTSTRAP.md, ClaudeInvocation, structured output schema — untouched. The photo is purely host-side presentation.
- Sending the photo on subsequent bootstrap turns or at bootstrap completion. The photo lands once, with the very first agent reply.

## Bot bootstrap photo flow

### Trigger predicate

In `crates/bot/src/telegram/worker.rs`, after a successful CC reply, send the photo if and only if BOTH:

1. `bootstrap_mode == true` — `BOOTSTRAP.md` exists in the agent dir. Already computed at worker.rs:1158.
2. The just-completed CC invocation **created** a new session (used `--session-id <uuid>`, not `--resume <root>`). This signals "first message in this chat".

The new-session-vs-resume decision is made when building the `ClaudeInvocation`; capture it as a local `bool first_turn_in_chat` and thread it to the post-reply branch. **No new SQLite query.**

Predicate factored into a tiny pure function for testability:

```rust
fn should_send_bootstrap_photo(bootstrap_mode: bool, first_turn_in_chat: bool) -> bool {
    bootstrap_mode && first_turn_in_chat
}
```

### Asset embedding

```rust
const BOOTSTRAP_WELCOME_PNG: &[u8] = include_bytes!(concat!(
    env!("CARGO_MANIFEST_DIR"),
    "/../../assets/character-on-coal.png"
));
```

`CARGO_MANIFEST_DIR` resolves to the `crates/bot/` directory, so the relative path goes up two levels to the repo root and then into `assets/`. Anchoring on the manifest dir keeps the embed working regardless of which file inside the crate triggers the macro. Compile-time bytes, no filesystem deploy, no codegen entry, no platform-store entry. The asset travels with the bot binary.

### Send call

```rust
use teloxide::types::InputFile;

let file = InputFile::memory(BOOTSTRAP_WELCOME_PNG.to_vec())
    .file_name("welcome.png");
let mut req = bot.send_photo(chat_id, file);
if let Some(thread) = thread_id {
    req = req.message_thread_id(thread);
}
let _ = req.await; // see Failure handling
```

### Send order

Photo **before** the text reply:

```
[user's first message arrives]
  → bot: spawn CC with --session-id <new uuid>, bootstrap_mode=true
  → CC produces greeting
  → bot: send_photo(chat_id, thread_id?, welcome.png)   // NEW
  → bot: send_text(chat_id, thread_id?, greeting)       // existing
```

Photo first means the chat reads: claw-with-eyes → "Hi, what should I call you?" — the avatar appears, then speaks.

### Thread / topic correctness

For Telegram group chats with topic threads, the photo goes to the same `(chat_id, thread_id)` as the text reply. Reuse the existing thread-id resolution (whatever the surrounding `send_text` path uses).

### Failure handling

- `send_photo` failures (network, Telegram API, oversized file, etc.) are wrapped in a `match`; on `Err`, `tracing::warn!(?err, "bootstrap welcome photo failed")` and continue. The text reply MUST still land. The photo is a flourish, not a contract.
- If `BOOTSTRAP_WELCOME_PNG` is somehow malformed (asset corrupted in repo), the bot still starts (compile-time embedding doesn't validate PNG content); the runtime call to Telegram fails and we log + continue. CI test (below) catches missing/corrupt assets before merge.

## Tests

- `assets/character-on-coal.png` non-empty + starts with PNG magic bytes (`89 50 4E 47 0D 0A 1A 0A`). Catches a missing or corrupted asset at CI time. Lives in the bot crate as a unit test next to the `include_bytes!`.
- `should_send_bootstrap_photo` table-driven: four rows covering the cartesian product of `bootstrap_mode × first_turn_in_chat`. Only `(true, true)` returns true.
- No integration test for the live `send_photo` call. The bot crate currently does not mock teloxide for outbound sends; consistent with that pattern.
